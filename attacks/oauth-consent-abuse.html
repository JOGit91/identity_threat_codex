OAuth Consent Abuse

Critical Persistence Credential Access T1550.001 T1528 Updated Jun 2025

OAuth consent abuse — also called illicit consent grant or OAuth phishing — tricks a user into granting a malicious application persistent, delegated access to their Microsoft 365 data. The access survives password resets, MFA changes, and session revocations because it operates through an OAuth refresh token bound to the application, not to the user's session.

What is OAuth Consent Abuse? 01

Microsoft 365 uses OAuth 2.0 to allow third-party applications to access user data — with the user's permission. When you connect an app like Salesforce, Zoom, or a productivity tool to your Microsoft account, you're presented with a consent screen listing the permissions the app is requesting: "Read your email," "Access your files," "Sign you in." If you click Accept, Microsoft issues the app a refresh token it can use to access your data on your behalf, indefinitely.

OAuth consent abuse exploits this legitimate mechanism. The attack works as follows:

  1. The attacker registers a malicious application in a Microsoft Entra ID tenant they control. The app is given a convincing display name: "M365 Security Update," "Microsoft Teams Integration," "IT Help Desk Portal."
  2. The attacker crafts an OAuth authorization URL specifying the malicious app's client ID and a list of high-value permission scopes (Mail.ReadWrite, Files.ReadWrite.All, User.Read.All).
  3. The link is sent to the victim — via phishing email, Teams message, LinkedIn, or SMS — with a pretext compelling them to click it.
  4. The victim is taken to the genuine Microsoft consent page at login.microsoftonline.com. They authenticate with their real credentials and MFA. Microsoft displays the consent prompt for the malicious app.
  5. If the victim clicks Accept, Microsoft issues the malicious app a refresh token. The attacker now has persistent, application-level access to the victim's data — with no further involvement from the victim required.
This Persists Through Everything

An OAuth app consent grant is not tied to a session, a password, or an MFA device. If the victim changes their password, the app's refresh token remains valid. If the victim revokes all sessions, the app's token remains valid. The only way to terminate access is to explicitly revoke the consent grant in Entra ID — which most users have no visibility into and no habit of checking.

User vs. Admin Consent

Microsoft 365 has two consent types: user consent (a single user can approve the app for their own data) and admin consent (a Global Admin approves the app for the entire organization). By default in many tenants, users can consent to apps requesting low-privilege permissions without any admin approval. This is the gap that consent abuse exploits — the victim is an ordinary user, and they have the authority to consent without IT ever seeing a notification.

Why Do Threat Actors Use It? 02

  • The access is persistent by design. OAuth refresh tokens for confidential clients can have lifetimes measured in months or years. The attacker maintains continuous access to the victim's mailbox and files without any further phishing, credential theft, or attacker-side activity that would trigger detection.
  • Password resets don't help. Security teams who detect a compromised account typically force a password reset as their first containment action. For OAuth consent abuse, this does nothing — the app's token is not tied to the user's password.
  • The consent page is Microsoft's real page. The victim is authenticating to a legitimate Microsoft URL. There is no fake login page to detect, no certificate warning, no phishing domain. Security gateways have nothing to block.
  • Abuse happens server-side, silently. Once consent is granted, the attacker's application silently calls the Microsoft Graph API from their infrastructure. There is no sign-in event in the victim's account from the attacker's IP — only application access events that blend into normal background noise.
  • Useful even with low privileges. Even Mail.Read (read-only mailbox access) is enormously valuable for business email compromise reconnaissance. The attacker reads months of email to understand payment processes, active vendors, and executive relationships — without needing elevated privileges.
  • Low technical barrier. Registering an Entra ID app requires only a free Microsoft 365 account and a few minutes in the Azure Portal. Crafting the consent URL is a matter of substituting your app's client ID into a template.

What Happens If Attackers Succeed? 03

Delegated Access by Scope

The impact depends on which permissions the attacker requested and the victim consented to. Common high-value scopes:

ScopeAccess GrantedAttacker Use
Mail.ReadWrite Read, send, delete, and organize email BEC reconnaissance, inbox rule creation, impersonation
Files.ReadWrite.All Full OneDrive and SharePoint file access Data exfiltration, ransomware staging
Contacts.ReadWrite Read and modify contacts Harvest targets for further phishing; map organizational relationships
User.ReadBasic.All Read names, emails, titles of all users Directory enumeration; target selection for BEC
offline_access Persistent refresh token (no expiry on access) Long-term silent access; this scope is almost always included in malicious consent requests

Why This Is Particularly Dangerous for BEC

Business Email Compromise attackers who gain Mail.ReadWrite consent on a finance or executive account can read months of email history silently, with no sign-in event on the user's record. They identify active payment workflows, vendor relationships, and pending wire transfers. When ready to strike, they create inbox rules to hide replies and initiate fraudulent wire transfer requests — all while the legitimate user continues working from their account with no indication of compromise.

Incident Response Blind Spot

When organizations discover a BEC incident and force a password reset, the attacker's OAuth app retains access. Incomplete remediation that misses consent grants is documented in multiple incident response reports as the reason re-compromise occurred within hours of the initial response.

Detection & Threat Hunting 04

Log Sources

Source What to Look For Table (Sentinel) License

Hunt Query 1 — Consent Grants to Unverified Publishers

KQL Microsoft Sentinel — AuditLogs 
// Consent grants where the app publisher is not verified by Microsoft
// Unverified publishers are a strong risk indicator for malicious apps
AuditLogs
| where TimeGenerated > ago(30d)
| where OperationName == "Consent to application"
| mv-expand TargetResources
| extend AppName = tostring(TargetResources.displayName)
| extend InitiatedBy = tostring(InitiatedBy.user.userPrincipalName)
| extend ModifiedProps = TargetResources.modifiedProperties
| mv-expand ModifiedProps
| extend PropName = tostring(ModifiedProps.displayName)
| extend PropValue = tostring(ModifiedProps.newValue)
| where PropName == "ConsentType"
| extend ConsentType = PropValue
| project
    TimeGenerated,
    InitiatedBy,
    AppName,
    ConsentType,
    Result,
    CorrelationId
| order by TimeGenerated desc

Hunt Query 2 — High-Privilege Consent Grants

KQL Microsoft Sentinel — AuditLogs 
// OAuth consent grants that include high-value mail, file, or directory permissions
AuditLogs
| where TimeGenerated > ago(90d)
| where OperationName == "Consent to application"
| mv-expand TargetResources
| mv-expand TargetResources.modifiedProperties
| extend PropName = tostring(TargetResources_modifiedProperties.displayName)
| extend PropValue = tostring(TargetResources_modifiedProperties.newValue)
| where PropName == "ConsentAction.Permissions"
| where PropValue has_any (
    "Mail.ReadWrite",
    "Mail.Send",
    "Files.ReadWrite.All",
    "User.ReadWrite.All",
    "Directory.ReadWrite.All",
    "full_access_as_user",
    "offline_access"
  )
| extend Initiator = tostring(InitiatedBy.user.userPrincipalName)
| project
    TimeGenerated,
    Initiator,
    PermissionsGranted = PropValue,
    Result
| order by TimeGenerated desc

Hunt Query 3 — Service Principal Accessing Mail After Recent Consent

KQL Microsoft Sentinel — AuditLogs + OfficeActivity 
// App (service principal) accessing mailbox within 24h of a consent event
// Indicates rapid exploitation of a freshly granted consent
let RecentConsents = AuditLogs
| where TimeGenerated > ago(30d)
| where OperationName == "Consent to application"
| mv-expand TargetResources
| project ConsentTime = TimeGenerated,
    AppName = tostring(TargetResources.displayName),
    ConsentedBy = tostring(InitiatedBy.user.userPrincipalName);
OfficeActivity
| where TimeGenerated > ago(30d)
| where Operation in ("MailItemsAccessed", "Send")
| where ClientAppId != ""  // Filter to application-level access (not user sessions)
| project AccessTime = TimeGenerated, UserId, ClientAppId, Operation
| join kind=inner RecentConsents on $left.UserId == $right.ConsentedBy
| where AccessTime between (ConsentTime .. (ConsentTime + 24h))
| project ConsentTime, AccessTime, UserId, AppName, Operation
MDCA App Governance is the Best Tool for This

Manual KQL hunting is important for existing compromises. For ongoing protection, Microsoft Defender for Cloud Apps App Governance provides a dedicated consent app dashboard with risk scoring, anomaly detection, and one-click app disabling. It surfaces high-risk apps immediately — including those with offline_access + high-privilege scopes consented by non-admin users.

Threat Actors 05

Midnight Blizzard (APT29)

Russia's SVR is documented using OAuth app registration as a persistence and lateral movement technique after initial access is obtained. After compromising a user via other means, Midnight Blizzard registers OAuth apps in the victim's tenant or uses existing apps to maintain persistent access that survives credential resets. Their 2023 campaign against Microsoft's own corporate environment involved malicious OAuth apps used to access source code repositories and internal email.

Storm-0062 and Similar BEC Actors

Financially motivated actors targeting business email compromise frequently combine credential phishing with OAuth consent phishing. After initial access, they pivot to registering malicious apps to create a persistent foothold in the mailbox that survives the inevitable password reset triggered when the victim or IT notices unusual activity.

Scatter Swine / Scattered Spider

Scattered Spider (tracked as UNC3944 / Octo Tempest) has used OAuth application abuse as part of their broader M365 persistence toolkit. Their initial access via Teams impersonation and vishing is often followed by OAuth app consent abuse to establish persistent access independent of any single compromised credential.

Commodity Threat Actors

Microsoft's 2023 detection of the "illicit consent grant" attack pattern documented a broad campaign of consent phishing targeting Microsoft 365 users with apps requesting Mail.Read and Contacts.Read permissions. The campaign was not attributed to a named actor — illustrating that this technique has diffused beyond nation-state operators into the broader threat actor ecosystem.

Follow-on Activity 06

  • Silent, long-term mailbox surveillance: With Mail.Read, the attacker's app polls the victim's inbox via Graph API. From the attacker's perspective, this is just an API call from their server — it produces no sign-in event in the victim's account and blends into normal application background traffic.
  • Inbox rule creation via Graph API: Using Mail.ReadWrite, the attacker creates forwarding or deletion rules through the API — which appear identically to user-created rules in audit logs and are caught by the same alert logic.
  • Data exfiltration: With Files.ReadWrite.All, the attacker bulk-downloads OneDrive and SharePoint files via Graph. This may appear as application-level file access rather than user-driven access, potentially bypassing user-focused DLP controls.
  • Token propagation to other users: If the consented app has User.ReadBasic.All or directory read permissions, the attacker can enumerate all users and send targeted consent phishing links to additional victims — expanding their foothold across the tenant.
  • Persistence across remediation: When a compromised account is discovered, the standard response (password reset, session revocation) leaves the OAuth app consent intact. The attacker retains access until the consent grant is explicitly revoked in Entra ID — which many incident response teams miss.

Prevention 07

The Primary Control: Restrict User Consent in Entra ID

Entra ID has a tenant-wide setting that controls what users can consent to without admin approval. The most secure setting is "Do not allow user consent" — all app consent requires an admin to approve. A practical middle ground is "Allow user consent for apps from verified publishers only" combined with a user consent workflow that routes requests to an admin for approval. This eliminates the ability to consent to apps from unverified publishers — the most common malicious app category.

Key Prevention Controls

  • Disable or restrict user consent in Entra ID — Entra ID admin center → Enterprise applications → Consent and permissions → User consent settings. Set to "Do not allow user consent" or "Allow for verified publishers with low-risk permissions" and enable the admin consent workflow.
  • Deploy MDCA App Governance — Provides full visibility into all OAuth apps consented in the tenant, with risk scoring, anomaly detection, and the ability to disable suspicious apps immediately without revoking the user's account.
  • Audit existing consent grants — Before locking down new consents, audit what's already there: Entra ID → Enterprise applications → All applications, filter by "User assigned" and sort by "Permissions." Any app with broad Mail or Files permissions from an unverified publisher warrants investigation.
  • Security awareness training — Train users to recognize the signs of a malicious consent prompt: unfamiliar app name, broad permissions (especially offline_access + Mail.ReadWrite), and a sense of urgency in the lure message.
  • Include consent revocation in incident response procedures — Every M365 incident response runbook should include a step to check for and revoke OAuth consent grants in Entra ID → Enterprise applications. Password reset alone is not sufficient containment.
  • Restrict Teams app permissions — Teams apps that request OAuth consent follow the same consent mechanism. An allowlist-only Teams app policy prevents malicious apps from being installed through the Teams channel.
