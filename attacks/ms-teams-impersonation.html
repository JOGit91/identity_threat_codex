MS Teams Impersonation

Critical Initial Access Social Engineering T1566 T1534 Updated Jun 2025

Microsoft Teams has become a primary social engineering vector for sophisticated threat actors. Attackers exploit the inherent trust users place in Teams messages — impersonating IT helpdesks, executives, and Microsoft Support to steal credentials, manipulate MFA approvals, and deliver malware directly inside an organization's collaboration environment.

What is MS Teams Impersonation? 01

Microsoft Teams is one of the most widely used enterprise communication platforms in the world, and it operates with a fundamentally different trust model than email. Users have been trained to be skeptical of email from external senders, but Teams messages — especially ones that appear to come from within the organization — are widely treated as implicitly trusted. Attackers exploit this gap.

MS Teams impersonation encompasses several distinct but related attack vectors:

  • External tenant messaging abuse: By default, Microsoft 365 tenants allow Teams messages from users in other Microsoft tenants. Attackers create Microsoft 365 trial accounts or use compromised tenants to send messages to victims that appear in their Teams client with an "(External)" label — which many users ignore or don't notice. The sender can set their display name to "Microsoft Support," "IT Help Desk," or the name of an internal employee.
  • Compromised internal account messaging: After compromising an internal user's account (via phishing, token theft, or credential stuffing), attackers use that account to send Teams messages to colleagues. These messages appear to come from a trusted internal source with no external label.
  • Malware delivery via Teams attachments: Teams allows file sharing within and across tenants. Attackers deliver malicious files (.exe, .vbs, weaponized Office documents, LNK files) directly to victims via Teams chat. Unlike email, Teams messages do not pass through most email security gateways.
  • Vishing / MFA manipulation via Teams calls: Attackers call victims via Teams (voice or video) impersonating IT helpdesk staff. While on the call, they walk the victim through steps that either install remote access tools or pressure them to approve MFA push notifications the attacker has triggered.
The "(External)" Label is Not Enough

Microsoft Teams does show an "(External)" designation for messages from outside the tenant. However, research and incident data consistently show that users treat this label as background information rather than a security warning — especially when the message content creates urgency (e.g., "Your account is about to be suspended"). This is a social engineering problem as much as a technical one.

Why Do Threat Actors Use It? 02

  • It bypasses email security controls entirely. Your organization may have Defender for Office 365, anti-phishing policies, DMARC, DKIM, and SPF all correctly configured — and none of those controls apply to Teams messages. A link delivered via Teams chat skips every email gateway inspection layer.
  • Users inherently trust Teams more than email. Years of phishing awareness training have conditioned people to be suspicious of email from external senders. Teams carries no such cultural suspicion. A Teams message from "IT Support" feels like an internal notification.
  • External messaging is enabled by default. Microsoft's default configuration allows any M365 user from any tenant to initiate a Teams chat with your users. Organizations that haven't explicitly restricted this are open to cold contact from external attackers.
  • Teams enables real-time social engineering. Unlike email phishing (asynchronous), Teams allows voice and video calls. An attacker can phone a victim, maintain live voice contact, create urgency and confusion, and walk them through malicious actions in real time.
  • Low barriers to impersonation. Anyone can create a Microsoft 365 trial account and set their display name to "Microsoft Support," "IT Help Desk," or the name of any internal employee found on LinkedIn. There is no domain or identity verification displayed to the recipient.
  • File delivery without detection. Many organizations have DLP and malware scanning on email but not on Teams. Attackers can deliver malicious files to an employee's Teams chat knowing the file may not be inspected.

What Happens If Attackers Succeed? 03

Credential Theft & Account Takeover

The most common outcome is credential theft. Attackers send victims a Teams message with a link to what appears to be a shared document, a security notification, or an IT portal. The link redirects to an AiTM phishing page or a credential harvesting site that mimics Microsoft's login portal. The victim enters their credentials and potentially approves an MFA request, giving the attacker full account access.

MFA Fatigue and Push Bombing

Attackers who already have a victim's credentials (from a separate data breach or credential dump) will initiate a Teams call impersonating IT helpdesk. While on the call, they trigger repeated MFA push notifications and pressure the victim to approve one — telling them "the system sent you an approval request, please accept it to verify your identity." This technique has proven extremely effective even against employees with security awareness training, particularly when attackers create urgency about account suspension or security incidents.

Malware Delivery

Storm-0324, a ransomware affiliate, industrialized Teams malware delivery by creating external Microsoft 365 tenants at scale and mass-delivering malicious file attachments (.exe, .lnk, .js) via Teams direct messages. Victims who opened files received DarkGate or Qakbot infections, which subsequently led to ransomware deployment. The attack chain bypassed email security entirely.

Remote Access Establishment

During vishing calls, attackers instruct victims to install remote access tools (QuickAssist, TeamViewer, AnyDesk, ScreenConnect) under the guise of IT troubleshooting. Once connected, the attacker has direct access to the victim's device, files, browser sessions, and can operate freely while the victim stays on the call.

Microsoft QuickAssist Abuse

Microsoft's built-in QuickAssist tool has been specifically abused by Black Basta affiliates in Teams-based attacks. Attackers impersonate internal IT staff via Teams, request screen share via QuickAssist, then install persistent backdoors. QuickAssist does not require administrative credentials to install or accept remote sessions.

Enable Teams Audit Logging

Microsoft 365 Teams audit events are captured in the Unified Audit Log, but you must ensure audit logging is enabled for your tenant (Compliance Center → Audit) and that you are ingesting OfficeActivity and CloudAppEvents into your SIEM. Without these, most Teams detection queries will return no data.

Hunt Query 1 — External Tenant Messaging Multiple Internal Users

KQL Defender XDR / Sentinel — CloudAppEvents 
// External tenant sending Teams messages to multiple internal users
// Possible mass phishing or social engineering campaign from external M365 tenant
// Tune the UniqueTargets threshold to your environment
CloudAppEvents
| where TimeGenerated > ago(7d)
| where Application == "Microsoft Teams"
| where ActionType has "Message"
| extend
    SenderDomain = tostring(parse_json(tostring(RawEventData)).SenderDomain),
    SenderUpn    = tostring(parse_json(tostring(RawEventData)).SenderUpn)
| where isnotempty(SenderDomain)
| where SenderDomain !endswith "yourdomain.com"   // Replace with your domain(s)
| summarize
    MessageCount  = count(),
    UniqueTargets = dcount(AccountUpn),
    Targets       = make_set(AccountUpn, 20),
    FirstSeen     = min(TimeGenerated),
    LastSeen      = max(TimeGenerated)
    by SenderDomain, SenderUpn, bin(TimeGenerated, 1h)
| where UniqueTargets >= 3
| order by UniqueTargets desc

Hunt Query 2 — External/Guest Users Sharing Files in Teams

KQL Defender XDR / Sentinel — CloudAppEvents 
// File uploads or shares from external/guest users via Teams
// Malware delivery vector — especially high-risk for executable file types
CloudAppEvents
| where TimeGenerated > ago(14d)
| where Application == "Microsoft Teams"
| where ActionType has_any ("FileUploaded", "FileSyncUploadedFull", "FileShared", "MessageCreatedHasAttachment")
| extend
    UserType = tostring(parse_json(tostring(RawEventData)).UserType),
    FileName = tostring(parse_json(tostring(RawEventData)).SourceFileName)
| where UserType in~ ("Guest", "External")
| extend FileExt = tostring(split(FileName, ".")[-1])
| where FileExt in~ (
    "exe","dll","vbs","js","bat","cmd","ps1","lnk","iso","zip","rar"  // High-risk extensions
  )
| project
    TimeGenerated,
    AccountDisplayName,
    AccountUpn,
    UserType,
    ActionType,
    FileName,
    FileExt,
    IPAddress
| order by TimeGenerated desc

Hunt Query 3 — External Teams Message Followed by High-Risk Sign-in

KQL Defender XDR / Sentinel — CloudAppEvents + SigninLogs 
// Correlate external Teams messages with subsequent high-risk sign-ins
// Suggests victim may have clicked a phishing link delivered via Teams
let ExternalMessages = CloudAppEvents
    | where TimeGenerated > ago(7d)
    | where Application == "Microsoft Teams"
    | where ActionType == "MessageCreatedHasLink"
    | extend SenderTenant = tostring(parse_json(tostring(RawEventData)).SenderTenantId)
    | where SenderTenant != ""       // Replace with your tenant ID
    | project MsgTime = TimeGenerated, TargetUpn = AccountUpn;
SigninLogs
| where TimeGenerated > ago(7d)
| where ResultType == 0
| where RiskEventTypes_V2 has_any (
    "unfamiliarFeatures",
    "anomalousToken",
    "anonymizedIPAddress",
    "impossibleTravel"
  )
| join kind=inner ExternalMessages
    on $left.UserPrincipalName == $right.TargetUpn
| where TimeGenerated between (MsgTime .. (MsgTime + 2h))
| project
    MsgTime,
    SigninTime  = TimeGenerated,
    UserPrincipalName,
    IPAddress,
    Location,
    RiskEventTypes_V2,
    AppDisplayName

Hunt Query 4 — Remote Access Tool Installation via Teams Context

KQL Defender XDR — DeviceProcessEvents 
// Remote access tools launched or installed on devices with recent Teams activity
// Indicative of vishing attack where IT impersonator requests remote session
// Requires Microsoft Defender for Endpoint
DeviceProcessEvents
| where TimeGenerated > ago(14d)
| where FileName in~ (
    "quickassist.exe",
    "anydesk.exe",
    "teamviewer.exe",
    "screenconnect.exe",
    "msra.exe",              // Remote Assistance
    "rustdesk.exe"
  )
| where InitiatingProcessFileName !in~ (
    "msiexec.exe",           // Exclude legitimate managed installs
    "setup.exe"
  )
| project
    TimeGenerated,
    DeviceName,
    AccountName,
    FileName,
    ProcessCommandLine,
    InitiatingProcessFileName,
    InitiatingProcessCommandLine
| order by TimeGenerated desc

Known Threat Actors 05

MS Teams impersonation has been documented across multiple distinct threat actor clusters, ranging from nation-state groups to financially motivated cybercriminal organizations.

Midnight Blizzard
APT29 · Cozy Bear · NOBELIUM
Nation-State (Russia)
In August 2023, Microsoft publicly disclosed that Midnight Blizzard conducted targeted Teams-based social engineering against fewer than 40 global organizations — primarily government agencies, NGOs, and tech companies. They created Microsoft 365 trial tenants and sent messages impersonating Microsoft technical support staff, attempting to steal MFA credentials via phishing links.
Scattered Spider
UNC3944 · Muddled Libra
Cybercriminal
Known for sophisticated helpdesk impersonation via Teams voice calls. They research targets thoroughly using LinkedIn and company websites, then call IT helpdesks or message employees pretending to be IT support personnel. They request MFA reset assistance or pressure victims into approving MFA push notifications while simultaneously attempting account takeovers.
Storm-0324
TA543 · Cyclops
Cybercriminal
Industrialized Teams-based malware delivery. Storm-0324 created large numbers of external Microsoft 365 tenants and sent Teams messages at scale containing malicious file attachments. Used to distribute DarkGate and Qakbot malware as an initial access broker, with payloads frequently leading to ransomware. Microsoft disrupted their infrastructure in 2023 but the technique remains active.
Black Basta (Affiliates)
SCATTERED SPIDER Adjacent
Cybercriminal
Black Basta ransomware affiliates were documented in 2024 using Teams vishing calls to impersonate IT helpdesk staff. They initiate contact via Teams, report fake security incidents requiring immediate resolution, and instruct victims to install QuickAssist or accept a remote session — deploying QAKBOT, Cobalt Strike, and ultimately Black Basta ransomware.

Typical Follow-on Activity 06

The follow-on chain differs by attack vector. Below are the two most common progressions.

Chain A: Phishing Link via Teams → Account Takeover

  1. External Teams message with urgent pretext ("Your account access will be suspended," "Verify your identity now")
  2. Victim clicks link → redirected to AiTM phishing page mimicking Microsoft login
  3. Credentials and MFA token stolen → attacker authenticates to M365 using stolen session cookie
  4. Post-compromise actions → inbox rules, data exfiltration, BEC (see Session Token Theft profile for details)

Chain B: Vishing / QuickAssist → Ransomware

  1. Pre-attack spam bombing (optional) → flood victim's inbox with thousands of emails to create confusion and a pretext for IT contact
  2. Teams call or message impersonating IT helpdesk: "We've detected an issue with your account. I'm calling to help you resolve it."
  3. Victim instructed to install QuickAssist or accept remote access session
  4. Attacker runs discovery commands via remote session: enumerates local admin rights, active directory membership, connected network drives
  5. Persistence established → scheduled tasks, registry run keys, or a second legitimate remote access tool installed silently
  6. Lateral movement → credential dumping (Mimikatz, LSASS), use of stolen credentials to pivot to domain controllers or additional workstations
  7. Ransomware deployment → data exfiltration followed by encryption across the network
Speed of Escalation

In documented Black Basta attacks, the time from initial Teams contact to ransomware deployment was measured in hours, not days. The combination of real-time social engineering via Teams voice calls and legitimate remote access tools makes this attack chain particularly fast and difficult to interrupt once it begins.

Prevention & Hardening 07

  • P1
    Restrict External Teams Access to Specific Trusted Tenants Only
    In the Microsoft Teams Admin Center under External Access, change the setting from "Allow all external domains" to "Allow only specific external domains" — then whitelist only the external organizations your company regularly collaborates with. This eliminates the ability of attackers to create arbitrary Microsoft 365 tenants and contact your users cold. This is the single highest-impact control for this attack.
  • P1
    Enable Safe Links for Microsoft Teams
    Microsoft Defender for Office 365 Plan 1 Safe Links can scan URLs sent in Teams messages and block access to known malicious sites. Configure under Microsoft Defender Portal → Email & Collaboration → Policies & Rules → Safe Links. Enable "Teams" in the policy scope. This provides URL inspection for one of the most common delivery mechanisms — malicious links sent in Teams chat.
  • P1
    User Awareness: IT Will Never Ask You to Approve MFA in Teams
    Run specific training that tells employees: (1) IT helpdesk will never initiate contact via Teams to ask you to approve an MFA notification you didn't trigger; (2) any Teams message requesting urgent credential or identity verification is a red flag; (3) employees should call the IT helpdesk back on a known number before acting on any Teams request involving account access. This social engineering attack relies on users not knowing these norms.
  • P1
    Restrict or Remove Microsoft QuickAssist from Managed Devices
    QuickAssist is a built-in Windows tool that has been specifically abused in Teams-based helpdesk impersonation attacks (Black Basta, Scattered Spider). If your IT team does not use QuickAssist as a legitimate remote support tool, remove it via Microsoft Intune or Group Policy. If you need remote support capability, use your managed remote access tool (not QuickAssist) and train employees to accept sessions only from that tool.
  • P2
    Block File Downloads from External/Guest Users in Teams
    In the Teams Admin Center under Guest Access and External Access policies, restrict the ability of external and guest users to share files in Teams chats. Also configure SharePoint external sharing settings to align. This limits the malware delivery vector used by Storm-0324 and similar actors.
  • P2
    Deploy Defender for Cloud Apps Policies for Teams Anomalies
    Configure Microsoft Defender for Cloud Apps (MDCA) activity policies to alert when: an external user sends messages to more than N internal users within a time window; a guest or external user shares files with high-risk extensions; or when a new external tenant initiates contact with users in sensitive departments (Finance, HR, Executive). MDCA provides the richest Teams activity telemetry available.
  • P2
    Enable Number Matching & Additional Context for MFA Push
    Entra ID's Authenticator App now supports number matching (victim must type a code shown on screen into their Authenticator app) and additional context (shows the app and location requesting approval). Both controls dramatically reduce MFA fatigue/push bombing success rates. Enable both under Entra ID → Security → Authentication Methods → Microsoft Authenticator. Number matching is on by default for new tenants — verify it is enabled for yours.
  • P3
    Configure Microsoft Purview Communication Compliance for Teams
    Microsoft Purview Communication Compliance can monitor Teams messages for policy violations, including messages containing keywords associated with social engineering (e.g., "wire transfer," "urgent payment," "approve this request"). Useful for detecting ongoing campaigns targeting finance and HR teams via Teams.
  • P3
    Review and Restrict App Permissions in Teams
    Attackers can also abuse the Teams app store to install malicious applications into your tenant if app governance is loose. In the Teams Admin Center, restrict which apps users can install (allow only Microsoft-verified apps or a specific approved list). Enable Microsoft Defender for Cloud Apps App Governance add-on for OAuth app visibility.
