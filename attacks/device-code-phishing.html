Device Code Phishing

Critical Credential Access Initial Access T1528 T1566 Updated Jun 2025

Device code phishing abuses the OAuth 2.0 device authorization flow — a legitimate Microsoft authentication mechanism designed for input-constrained devices — to trick users into handing attackers a fully authenticated token. MFA is completed by the victim against Microsoft's real servers, and the attacker receives a valid access and refresh token without ever touching the login page.

What is Device Code Phishing? 01

The OAuth 2.0 Device Authorization Grant (RFC 8628) was designed for smart TVs, printers, and other devices that can't display a browser or accept text input efficiently. The flow works like this: the device requests a short code from the authorization server, displays it to the user, and asks them to visit a URL on another device (like a phone or laptop) to approve the authentication. Microsoft 365 supports this flow at microsoft.com/devicelogin.

Attackers have weaponized this legitimate flow. The attack sequence is straightforward:

  1. The attacker calls Microsoft's device code endpoint, specifying an OAuth client ID (often a well-known Microsoft app like Microsoft Office or Azure CLI).
  2. Microsoft returns a user_code (e.g., ABCD-EFGH) and a verification URI (microsoft.com/devicelogin).
  3. The attacker sends the victim a convincing message — via email, Teams, LinkedIn, or SMS — instructing them to visit the real Microsoft URL and enter the code. The pretext is typically an IT action, software activation, or security verification.
  4. The victim navigates to the genuine Microsoft login page, authenticates with their real credentials, and completes MFA.
  5. The attacker, who has been polling Microsoft's token endpoint since step 1, immediately receives an access token and refresh token for the victim's account.
The Critical Insight: The Victim Does Everything Right

The victim visits a real Microsoft URL. They enter their real password. They complete their real MFA. From the victim's perspective, nothing is wrong — there is no fake login page, no certificate warning, no unusual redirect. The attacker gets a legitimate token because the victim genuinely authenticated.

The attack is also persistent by design. Microsoft issues both an access token (valid 60–90 minutes) and a refresh token (valid up to 90 days). The attacker can silently mint new access tokens for months without any further victim interaction.

Why Do Threat Actors Use It? 02

  • MFA is not bypassed — it's completed. Unlike AiTM phishing, there is no proxy in the middle. The victim genuinely satisfies MFA. This means phishing-resistant MFA (FIDO2) offers no protection — the binding is to Microsoft's legitimate domain, which is exactly where the victim authenticates.
  • No infrastructure required. AiTM attacks require a reverse proxy server, domain, TLS certificate, and a phishing kit. Device code phishing requires only the ability to send a message and a terminal to run a single API call. The barrier to entry is extremely low.
  • The URL is real. The victim visits microsoft.com/devicelogin — an authentic Microsoft domain. Email gateways, safe links, and browser warnings have nothing to flag. There is no phishing domain to block.
  • Works cross-platform. The attack works regardless of whether the victim uses Windows, macOS, iOS, or Android. The victim is just visiting a web page.
  • Long-lived tokens. Refresh tokens are valid for up to 90 days. The attacker can maintain silent access for months without repeat interaction with the victim.
  • Widely documented in nation-state campaigns. Russian threat actors — notably Midnight Blizzard (APT29) and Star Blizzard — have used device code phishing extensively against government targets, think tanks, and NGOs. Tooling and tradecraft are well-established.

What Happens If Attackers Succeed? 03

The scope of access depends on which OAuth scopes the attacker requested when initiating the device code flow. Common scopes used in attacks include https://graph.microsoft.com/.default — granting broad Microsoft Graph access — which translates to:

Immediate Access

  • Exchange Online (email): Full read access to the victim's mailbox via Microsoft Graph. Attackers immediately query for emails containing keywords like "wire transfer," "invoice," "credentials," and "password."
  • OneDrive & SharePoint: Read access to all files the victim has permission to access. Intellectual property, financial records, HR data, and client information are all reachable via Graph API calls.
  • Microsoft Teams: Access to Teams messages and channels the victim can read. Allows reconnaissance of organizational structure, active projects, and internal communications without leaving visible trace in most logging configurations.
  • Entra ID (if scoped): Read access to directory objects, users, groups, and role assignments — valuable for privilege escalation targeting.

Persistence Without the Victim

  • Silent token refresh: The refresh token silently obtains new access tokens for up to 90 days. No sign-in event visible to the user, no MFA prompt, no opportunity for detection via normal authentication monitoring.
  • New device registration: Using the Graph API with the token, attackers can register a new Entra-joined device, obtaining a device-bound token that survives a password reset.
  • Additional MFA method registration: If MFA registration is not restricted by Conditional Access, attackers can add their own authenticator to the account.
  • OAuth app consent: The attacker can use the token to consent to additional OAuth applications, creating multiple persistent access paths.
Password Reset Does Not Terminate Access

A refresh token issued before a password reset remains valid unless explicitly revoked. Incident responders must call Revoke-MgUserAuthenticationMethod or use the Entra ID portal to revoke all refresh tokens as part of containment. Password reset alone is insufficient.

Hunt Query 1 — Identify Device Code Flow Sign-ins

KQL Microsoft Sentinel — SigninLogs 
// Identify successful sign-ins using device code flow
// Device code sign-ins should be rare in most orgs — any unexpected use warrants investigation
SigninLogs
| where TimeGenerated > ago(30d)
| where AuthenticationProtocol == "deviceCode" or ClientAppUsed == "Device Login"
| where ResultType == 0  // Successful only
| project
    TimeGenerated,
    UserPrincipalName,
    AppDisplayName,
    ClientAppUsed,
    IPAddress,
    LocationDetails,
    DeviceDetail,
    AuthenticationProtocol,
    TokenIssuerType
| order by TimeGenerated desc

Hunt Query 2 — Silent Token Refresh from Unexpected Location

KQL Microsoft Sentinel — AADNonInteractiveUserSignInLogs 
// Non-interactive token refreshes from a country not seen in the user's interactive sign-ins
// This detects attacker-side token refresh that is geographically inconsistent with the victim
let InteractiveCountries = SigninLogs
| where TimeGenerated > ago(30d)
| where ResultType == 0
| summarize KnownCountries = make_set(tostring(LocationDetails.countryOrRegion))
    by UserPrincipalName;
AADNonInteractiveUserSignInLogs
| where TimeGenerated > ago(7d)
| where ResultType == 0
| extend Country = tostring(LocationDetails.countryOrRegion)
| join kind=leftouter InteractiveCountries on UserPrincipalName
| where not(set_has_element(KnownCountries, Country))
| project
    TimeGenerated,
    UserPrincipalName,
    AppDisplayName,
    IPAddress,
    Country,
    KnownCountries
| order by TimeGenerated desc

Hunt Query 3 — Device Code Authentication Followed by Inbox Rule Creation

KQL Microsoft Sentinel — SigninLogs + OfficeActivity 
// Device code sign-in followed by suspicious inbox rule within 1 hour
// High-confidence indicator of post-compromise activity
let DeviceCodeSignins = SigninLogs
| where TimeGenerated > ago(30d)
| where AuthenticationProtocol == "deviceCode"
| where ResultType == 0
| project SigninTime = TimeGenerated, UserPrincipalName, IPAddress;
OfficeActivity
| where TimeGenerated > ago(30d)
| where Operation in ("New-InboxRule", "Set-InboxRule")
| where Parameters has_any ("DeleteMessage", "ForwardTo", "RedirectTo", "MoveToFolder")
| project RuleTime = TimeGenerated, UserId, Operation, Parameters
| join kind=inner DeviceCodeSignins on $left.UserId == $right.UserPrincipalName
| where RuleTime between (SigninTime .. (SigninTime + 1h))
| project SigninTime, RuleTime, UserId, IPAddress, Operation, Parameters
Baseline First

In most enterprise tenants, the device code flow is legitimately used by Azure CLI, some PowerShell modules, and developer tools. Before alerting on all device code sign-ins, run Hunt Query 1 across 30 days to establish your baseline. Alert on deviations from that baseline — new apps, new countries, unexpected user populations.

Threat Actors 05

Midnight Blizzard (APT29 / Cozy Bear)

Russia's SVR foreign intelligence service. Microsoft and industry researchers documented Midnight Blizzard using device code phishing against government agencies, NGOs, think tanks, and research institutions across Europe and North America. Their campaigns are meticulously crafted — impersonating Microsoft support, security teams, or IT administrators via email and WhatsApp. The pretext often references an urgent security action, multi-factor authentication configuration, or software update.

Star Blizzard (SEABORGIUM / Callisto Group)

FSB-linked threat actor targeting Western government officials, journalists, academics, and defense sector organizations. Star Blizzard has extensively used device code phishing in credential harvesting campaigns, often preceded by extended spear-phishing reconnaissance. They frequently impersonate colleagues, conference organizers, or known contacts to increase the likelihood the victim completes the device code entry.

Storm-2372

A threat cluster tracked by Microsoft using device code phishing at scale against a broad range of organizations in Europe and North America. Active campaigns in 2024–2025 used Teams, WhatsApp, Signal, and email as delivery channels. The volume and targeting breadth suggest a financially motivated or broad espionage mandate rather than highly targeted intelligence collection.

Financially Motivated Actors (BEC)

Device code phishing is increasingly appearing in Business Email Compromise operations outside of nation-state contexts. The technique's low infrastructure requirements make it accessible to less sophisticated actors. BEC actors use it to gain access to finance department mailboxes and SharePoint documents for wire fraud operations.

Follow-on Activity 06

After obtaining a token via device code phishing, documented follow-on activity includes:

  • Email reconnaissance: Search of the mailbox for keywords related to financials, credentials, VPN access, and organizational structure using Graph API calls that appear as background application access, not user-driven sign-ins.
  • Inbox rule creation: Forwarding rules to attacker-controlled mailboxes; deletion rules targeting security alert emails.
  • Device registration: Registering a new Entra ID-joined device using the token, obtaining a persistent hardware-bound credential.
  • OAuth app consent: Using the token to consent additional malicious OAuth applications, creating redundant access paths that survive password resets and token revocation.
  • SharePoint / OneDrive enumeration: Bulk download of accessible documents for exfiltration or further targeting.
  • Lateral movement via email: Sending device code phishing lures to the victim's contacts from the compromised account, extending the campaign to additional targets with the added credibility of a known sender.

Prevention 07

The Primary Control: Block Device Code Flow in Conditional Access

Entra ID Conditional Access has an Authentication Flows condition (generally available as of 2024) that explicitly allows or blocks the device code flow. For most organizations where end users do not need the device code flow, blocking it entirely for non-privileged users eliminates the attack surface at the source. This requires Entra ID P1 licensing.

Key Prevention Controls

  • Block device code flow via Conditional Access Authentication Flows — The single most effective technical control. Create a CA policy targeting all users, set Authentication Flow condition to Device Code Flow, and block. Scope exclusions for any legitimate service accounts or admin workflows that require it.
  • Security awareness training — The only control that works regardless of what flow is being abused. Train users: never enter a device code at microsoft.com/devicelogin unless you personally initiated a sign-in on a device that asked you to go there. Urgency, unsolicited requests, and IT impersonation are red flags.
  • Token lifetime policies — Reduce refresh token lifetimes for non-compliant scenarios. Enforcing shorter sign-in frequencies means attackers must re-authenticate more often, increasing detection opportunities.
  • Continuous Access Evaluation — Ensures that once the compromise is detected and the account is remediated, active tokens are revoked in near-real-time rather than waiting out the full lifetime.
  • Entra ID Protection — The Anomalous Token and Unfamiliar Sign-in Properties detections catch post-issuance anomalous usage. Combined with a Sign-in Risk CA policy, this provides automated response when unusual token activity is detected.
  • Restrict MFA method registration — Block the most common follow-on persistence action: registering a new authenticator from an untrusted location post-compromise.
