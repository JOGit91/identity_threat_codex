Session Token Theft

Critical Credential Access Defense Evasion T1539 T1550.001 Updated Jun 2025

Session token theft — also known as pass-the-cookie, AiTM (Adversary-in-the-Middle), or token hijacking — allows attackers to steal authenticated session tokens after a victim has already passed MFA, granting them full access to Microsoft 365 services without ever knowing the user's password.

What is Session Token Theft? 01

Modern web applications — including every Microsoft 365 service — authenticate users using session tokens: short-lived cryptographic strings that are issued after a successful login. Once issued, these tokens act as proof of identity and are presented with every subsequent request. This is what keeps you logged in to Outlook or SharePoint without being asked for your password every few minutes.

The critical insight attackers exploit is this: a valid token is as good as a valid login. If an attacker can steal your session token, they don't need your password, they don't need to know your MFA method, and they don't need to solve any authentication challenge. They simply replay the stolen token to Microsoft's servers, which respond exactly as they would to you — granting full authenticated access.

Token theft happens through several delivery mechanisms:

  • Adversary-in-the-Middle (AiTM) phishing: The attacker stands up a reverse proxy (using tools like Evilginx2, EvilProxy, or Modlishka) that sits between the victim and the real Microsoft login page. The victim authenticates — including completing MFA — against the real Microsoft server, but all traffic flows through the attacker's proxy, which captures the resulting session cookie.
  • Malware-based extraction: Infostealer malware (Redline, Raccoon, Vidar) running on a victim's device extracts session cookies directly from browser storage, where they are stored in plaintext or easily decryptable formats.
  • Browser extension abuse: Malicious browser extensions can read and exfiltrate cookies for any domain the victim visits.
  • Token theft from memory / disk: On corporate devices, tokens may be cached in MSAL (Microsoft Authentication Library) token caches, which can be extracted by attackers with local access.
Why MFA Doesn't Stop This

MFA protects the authentication event. Once authentication succeeds and a token is issued, MFA has already done its job. A stolen token carries the proof that MFA was already satisfied — so the attacker never needs to trigger another MFA challenge.

Why Do Threat Actors Use It? 02

  • It completely bypasses MFA. This is the primary draw. The vast majority of M365 security guidance tells organizations to enable MFA — and that advice is correct. But token theft exploits the gap between authentication and authorization, rendering MFA irrelevant once a session is established.
  • Tokens are long-lived. Microsoft 365 refresh tokens can be valid for up to 90 days by default. Even access tokens (shorter-lived, typically 60-90 minutes) can be used immediately for high-value actions. Refresh tokens can silently mint new access tokens without the user's involvement.
  • No noisy credential attacks. Brute force and password spraying generate failed authentication attempts that are easy to alert on. Token theft produces successful sign-ins that look legitimate.
  • Tooling is widely available. Evilginx2, EvilProxy, Modlishka, and Muraena are all open-source or commercially available AiTM toolkits. They require minimal technical skill to deploy and can be had for a few hundred dollars in underground markets.
  • Works against all M365 services. A stolen Microsoft session token grants access to Exchange Online, SharePoint, Teams, OneDrive, Azure Portal, and any other service the victim has access to under the same tenant.
  • High ROI for BEC campaigns. Business Email Compromise is one of the highest-ROI fraud categories in cybercrime. A single successful token theft from a finance employee or executive can yield hundreds of thousands of dollars in wire fraud.

What Happens If Attackers Succeed? 03

Immediate Access

  • Exchange Online (email): Full read/write access to the victim's inbox. Attackers immediately search for keywords like "wire transfer," "invoice," "password," "CEO," and "bank account."
  • SharePoint & OneDrive: Access to all documents the victim can reach — including shared drives, financial records, legal documents, HR data, and client information.
  • Microsoft Teams: Access to all Teams conversations, channels, and files. Attackers can read historical messages and use the victim's identity to send messages to colleagues.
  • Azure Portal / Entra ID: If the victim is a privileged user (Global Admin, Application Admin, etc.), the attacker gains access to the entire tenant, including the ability to add new admin accounts.

Persistence & Escalation

  • Inbox rules: Attackers immediately create forwarding or deletion rules to hide security alert emails and forward copies of email to external addresses.
  • New device registration: Adding a new Entra ID-joined or registered device allows the attacker to obtain fresh tokens tied to a device they control, surviving a password reset.
  • New MFA methods: If Entra ID Protection policies allow, attackers register new authenticator apps or phone numbers, locking the victim out.
  • OAuth app consent: Granting a malicious OAuth application consent to the victim's account creates persistent token-based access that survives password resets.
Password Reset Does Not Revoke Tokens by Default

If a victim's password is reset without also revoking all active sessions in Entra ID, existing stolen tokens may remain valid. Always use Revoke-AzureADUserAllRefreshToken (or the Entra portal equivalent) as part of incident response.

Financial & Downstream Impact

  • Wire fraud via email impersonation of the compromised account
  • Invoice manipulation targeting finance teams and vendors
  • Data exfiltration for extortion or sale
  • Supply chain compromise: using the victim's account to attack their customers and partners
  • Ransomware deployment following lateral movement within the tenant

Prerequisite: Enable Entra ID Protection

The single most impactful detection for token theft is the "Anomalous Token" risk event in Entra ID Protection. It detects tokens being used from unusual locations, devices, or with anomalous claims. This requires an Entra ID P2 (or Microsoft 365 E5 / E5 Security) license.

Hunt Query 1 — Anomalous Token Risk Events

KQL Microsoft Sentinel — AADUserRiskEvents 
// Entra ID Protection: Anomalous Token risk events
// Requires Entra ID P2 license. High-fidelity signal for token theft.
AADUserRiskEvents
| where TimeGenerated > ago(7d)
| where RiskEventType == "anomalousToken"
| project
    TimeGenerated,
    UserPrincipalName,
    RiskLevel,
    RiskDetail,
    IpAddress,
    Location,
    AdditionalInfo
| order by RiskLevel asc, TimeGenerated desc

Hunt Query 2 — Impossible Travel & Risk Events in Sign-in Logs

KQL Microsoft Sentinel — SigninLogs 
// Sign-ins flagged with high-risk signals
// Catches token replay, impossible travel, and unfamiliar properties
SigninLogs
| where TimeGenerated > ago(7d)
| where ResultType == 0                       // Successful sign-ins only
| where RiskEventTypes_V2 has_any (
    "anomalousToken",
    "impossibleTravel",
    "unfamiliarFeatures",
    "anonymizedIPAddress",
    "maliciousIPAddress"
  )
| project
    TimeGenerated,
    UserPrincipalName,
    AppDisplayName,
    IPAddress,
    Location,
    RiskEventTypes_V2,
    AuthenticationRequirement,
    IsCompliant,
    DeviceDetail,
    ConditionalAccessStatus
| order by TimeGenerated desc

Hunt Query 3 — Suspicious Inbox Rules Post Sign-in

KQL Microsoft Sentinel — OfficeActivity 
// Inbox rules with attacker-typical parameters
// Attackers create these immediately after access to hide security alerts
// and forward email copies to external addresses
OfficeActivity
| where TimeGenerated > ago(14d)
| where Operation in ("New-InboxRule", "Set-InboxRule")
| extend Params = todynamic(Parameters)
| mv-expand Params
| where Params.Name in~ (
    "DeleteMessage",
    "ForwardTo",
    "RedirectTo",
    "MoveToFolder",
    "ForwardAsAttachmentTo",
    "MarkAsRead"
  )
| project
    TimeGenerated,
    UserId,
    ClientIP,
    Operation,
    RuleName = tostring(Params.Name),
    RuleValue = tostring(Params.Value)
| order by TimeGenerated desc

Hunt Query 4 — MFA "Previously Satisfied" on Non-Compliant Devices

KQL Microsoft Sentinel — SigninLogs 
// Token replay indicator: MFA satisfied by prior session
// on a device not recognized as compliant.
// Legitimate replay is normal; flag when combined with new/unknown device.
SigninLogs
| where TimeGenerated > ago(7d)
| where ResultType == 0
| where AuthenticationDetails has "Previously satisfied"
| where IsCompliant == false or IsCompliant == ""
| extend DeviceId = tostring(DeviceDetail.deviceId)
| where isempty(DeviceId)                     // No device identity — suspicious
| project
    TimeGenerated,
    UserPrincipalName,
    AppDisplayName,
    IPAddress,
    Location,
    AuthenticationRequirement,
    ConditionalAccessStatus,
    DeviceDetail
| order by TimeGenerated desc

Known Threat Actors 05

The following threat actors have been publicly documented using session token theft and AiTM techniques against Microsoft 365 environments.

Scattered Spider
UNC3944 · Muddled Libra · 0ktapus
Cybercriminal
Prolific English-speaking cybercriminal group with extensive AiTM phishing operations. Known for calling IT helpdesks while simultaneously performing token theft, and for bypassing Okta and Entra ID MFA at scale. Responsible for major attacks on MGM Resorts, Caesars, and numerous financial and technology firms.
Midnight Blizzard
APT29 · Cozy Bear · NOBELIUM
Nation-State (Russia)
Russian SVR-linked group known for sophisticated, persistent intrusions. In the 2024 Microsoft breach, used password spraying followed by legacy token abuse to compromise Microsoft corporate accounts. Routinely uses OAuth application-based token persistence across M365 tenants.
Storm-1167
Caffeine · Various BEC Groups
Cybercriminal
Operates a large-scale AiTM-as-a-service phishing platform sold to other criminal actors. Primarily motivated by Business Email Compromise and financial fraud. Targets Microsoft 365 accounts at volume; documented in Microsoft Threat Intelligence publications from 2022–2024.
EvilProxy Operators
Commercial AiTM Platform
Cybercriminal
EvilProxy is a commercial reverse-proxy phishing platform specifically designed to steal session cookies from M365, Google, and other services. Sold via dark web forums and Telegram. Used by many independent threat actors; the platform abstracts away the technical complexity of AiTM attacks.

Typical Follow-on Activity 06

Once an attacker has a valid session token, there is a predictable playbook they follow. The speed of this progression is alarming — in documented incidents, attackers complete steps 1–5 within minutes of gaining access.

  1. Immediate reconnaissance — Search inbox for keywords ("wire transfer," "password," "bank," "invoice," "CEO," "ACH"). Download the Global Address List. Enumerate SharePoint sites and document libraries.
  2. Create inbox rules — Set rules to delete or move security alert emails (from Microsoft, your SIEM, your IT team) to Deleted Items or a hidden folder. Optionally set up forwarding to an external account.
  3. Establish persistence — Register a new device in Entra ID, add a new MFA authenticator, or grant an OAuth application consent to maintain access even after a password reset.
  4. Business Email Compromise (BEC) — Impersonate the victim to contact finance teams, vendors, or executives. Request urgent wire transfers, invoice payment changes, or W-2/payroll diversions.
  5. Data exfiltration — Bulk download of SharePoint documents, Teams exports, OneDrive contents. Particularly interested in M&A activity, financial forecasts, intellectual property, and client data.
  6. Lateral movement — Use the victim's identity to pivot: phish colleagues via Teams or email, access shared mailboxes, or — if the victim is an admin — make tenant-level configuration changes.
  7. Extortion or sale — Exfiltrated data may be held for ransom, sold on dark web markets, or used to enable further attacks against the victim's customers and partners.
Supply Chain Risk

A compromised M365 account is a trusted launchpad for attacking everyone that account communicates with. Outbound phishing from a trusted domain has dramatically higher click rates. Threat actors use compromised accounts to target partners, customers, and vendors — multiplying the blast radius of a single token theft.

Prevention & Hardening 07

Prevention is layered. P1 items directly prevent token theft or limit its impact. P2 items detect and respond faster. P3 items reduce what an attacker can do once inside.

  • P1
    Deploy Phishing-Resistant MFA (FIDO2 / Windows Hello for Business)
    FIDO2 security keys and Windows Hello for Business are bound to the specific origin (domain) at registration. An AiTM proxy cannot replay or forward these credentials because the cryptographic proof is origin-locked. This is the only MFA method that stops AiTM phishing by design. Roll out to privileged accounts immediately, then all users.
  • P1
    Enable Conditional Access Token Protection (Token Binding)
    Token Protection binds access tokens to the specific device and session where they were issued. A token stolen and replayed from a different device will be rejected by Entra ID. Configure via Conditional Access → Session controls → "Require token protection for sign-in sessions." Currently in GA for specific apps; coverage expanding.
  • P1
    Enable Entra ID Protection & Sign-in Risk Policies (P2)
    Configure risk-based Conditional Access policies that require step-up authentication or block access when Entra ID Protection detects "Anomalous Token," "Impossible Travel," or "High" user risk. This is the primary real-time detection and enforcement mechanism for token theft. Requires Entra ID P2 or M365 E5.
  • P1
    Enable Continuous Access Evaluation (CAE)
    CAE allows Entra ID to revoke tokens in near-real-time (within minutes) when a critical event occurs — such as a password reset, account disable, or IP change that violates policy. Without CAE, stolen access tokens remain valid until they naturally expire. CAE is available at no extra cost but must be enabled in Conditional Access.
  • P2
    Configure Sign-in Risk Conditional Access Policies
    Create Conditional Access policies that block or require re-authentication when sign-in risk is Medium or High. Scope to all users and all cloud apps. Even without P2, consider Named Locations to block sign-ins from countries you don't operate in.
  • P2
    Reduce Token Lifetimes & Enable Strict Reauthentication
    In Conditional Access session controls, reduce the sign-in frequency for sensitive apps and privileged users. Disabling "Persistent Browser Session" forces reauthentication after browser closure, reducing the window a stolen token is useful. Balance usability — apply strict settings to privileged accounts first.
  • P2
    Restrict MFA Method Registration with Conditional Access
    Require that MFA registration only occurs from compliant, Entra ID-joined devices, or require re-verification of existing MFA before adding new methods. This prevents attackers who have gained token access from registering new authenticators to maintain persistence through a password reset.
  • P2
    Alert on New Inbox Rules with Suspicious Parameters
    Configure Sentinel or Defender alerts to fire when inbox rules are created with DeleteMessage, ForwardTo, or RedirectTo parameters. This is almost always the first detectable post-compromise action attackers take, and catching it fast limits damage.
  • P3
    Require Compliant Device for High-Sensitivity Apps
    Conditional Access policies that require device compliance or Hybrid Azure AD join for access to Exchange Online and SharePoint significantly increase the friction for attackers using stolen tokens from personal or unmanaged devices.
  • P3
    Block Legacy Authentication Protocols
    Legacy protocols (IMAP, POP3, SMTP AUTH, basic auth) do not support MFA or Conditional Access and are frequently abused for token-independent credential access. Block them in Exchange Online and Conditional Access. Microsoft has deprecated basic auth for most services, but verify your tenant configuration.
